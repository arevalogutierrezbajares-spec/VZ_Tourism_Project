# F3 — Security Audit
**Date:** 2026-04-20
**Branch:** main
**Auditor:** Claude Sonnet 4.6
**Risk Level:** High

---

## Executive Summary

The platform has several serious security weaknesses. Admin backend access is gated client-side by a SHA-256 hash that is hardcoded and publicly visible in the browser bundle, meaning anyone can brute-force or crack the password offline. The WhatsApp webhook now has proper HMAC-SHA256 signature verification and the Stripe webhooks are correctly secured. The most critical remaining issues are: the hardcoded `PMS_BRIDGE_SECRET` fallback in source code, the upload endpoint accepting arbitrary Supabase bucket names from user input, the geocode endpoint accepting an arbitrary `country` query parameter that could be used to probe Google Places beyond Venezuela, and the absence of rate limiting on the booking creation, Stripe checkout, and Google Places proxy endpoints. The platform's overall posture is adequate for a private beta but has multiple gaps that must be addressed before handling real payment volume.

## API Key / Secret Exposure

### VERIFIED — Google Maps API key is server-side only (known issue DOES NOT reproduce)
The audit task stated the key was exposed client-side in `app/(tourist)/explore/page.tsx` and `app/(tourist)/listing/[slug]/page.tsx`. After reading both files, **no Google Maps key is present in either**. The explore page is a pure server component using local data; the listing page uses only Supabase and `lib/local-listings`. The `GOOGLE_PLACES_API_KEY` is only used server-side in:
- `app/api/places/photo/route.ts` (line 23)
- `app/api/places/autocomplete/route.ts` (line 5)
- `app/api/ruta/geocode/route.ts` (line 15)
- `lib/google-places.ts` (line 1)

All four are server-only files. **The seeded issue does not exist in the current codebase.**

### FINDING — Hardcoded PMS bridge secret with weak fallback
**File:** `app/api/pms/[...path]/route.ts` line 6
```ts
const PMS_BRIDGE_SECRET = process.env.PMS_BRIDGE_SECRET || 'vav-bridge-dev-secret-2026';
```
The fallback value `'vav-bridge-dev-secret-2026'` is committed to source and will be used in production if the env var is not set. Any attacker who reads the source code (GitHub exposure, contractor access) can impersonate a valid VAV user to the PMS microservice.

### FINDING — Admin password hash committed to source code
**File:** `app/(admin)/admin/layout.tsx` line 29
```ts
const ADMIN_HASH = '1967e7168eb580368fcc78611a7ae30b7cb2b2938f1e2d2c865c48c719f69792';
```
This SHA-256 hash is a client-side bundle constant. Any visitor can extract it from DevTools and run it through a rainbow table (crackstation.net, hashcat) to recover the plaintext password offline.

### FINDING — `NEXT_PUBLIC_MAPBOX_TOKEN` exposed to browser
Used in `components/map/MapContainer.tsx` (line 320), `components/listing/ListingMap.tsx` (line 22), `lib/mapbox/helpers.ts`, and `lib/ruta/pricing.ts`. Mapbox public tokens are designed to be browser-visible, but must be restricted to your domain via the Mapbox dashboard token scope settings. If unrestricted, the token can be stolen and used to bill your account.

### FINDING — `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` in browser
**File:** `lib/stripe/client.ts` line 7. This is correct and intended behaviour for Stripe. No issue.

### NOT AN ISSUE — `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`
By Supabase design these are intended to be public. Row-level security must enforce access control.

### NOT AN ISSUE — `STRIPE_SECRET_KEY` / `STRIPE_WEBHOOK_SECRET`
Both are server-only (`lib/stripe/server.ts`, `process.env.STRIPE_SECRET_KEY`). Not in any NEXT_PUBLIC_ var.

### NOT AN ISSUE — `ANTHROPIC_API_KEY`
Server-only in all AI routes. No NEXT_PUBLIC_ prefix.

## Authentication & Authorization Gaps

### P0 — Admin layout guard is client-side only (broken auth model)
**File:** `app/(admin)/admin/layout.tsx`

The admin password check runs entirely in client-side JavaScript (`'use client'`). The flow is:
1. User types password → SHA-256 hash computed in browser (line 73)
2. Hash compared to `ADMIN_HASH` constant (line 74) — also in the browser bundle
3. Cookie set in browser if match (line 75)

**Impact:** Anyone can set `document.cookie = 'admin_token=1967e7168eb580368fcc78611a7ae30b7cb2b2938f1e2d2c865c48c719f69792; path=/'` in DevTools and gain full admin UI access without knowing the password. This bypasses the entire gate.

However, all admin API routes (`app/api/admin/**`) use `requireAdmin()` from `lib/api/require-auth.ts`, which calls `supabase.auth.getUser()` and checks `users.role === 'admin'` in the database (lines 26-44). This means the **UI is bypassed but the API is still protected** by real Supabase auth. The admin UI will render, but all data-mutating actions will return 403 unless the user is a legitimate admin in Supabase.

**Residual risk:** The admin UI renders with the forged cookie, and read-only views may surface sensitive business data if any admin GET endpoint lacks `requireAdmin()` verification.

### P1 — RUTA dispatch layout is client-side only (same broken model)
**File:** `app/(ruta)/ruta/dispatch/layout.tsx`

Uses `createBrowserClient` and `supabase.auth.getUser()` client-side to check `app_metadata.ruta_role`. This is a UI gate only. **However**, all dispatch API routes use `requireRutaRole()` server-side (verified in `app/api/ruta/dispatch/route.ts` line 6 and `app/api/ruta/dispatch/assign/route.ts` line 5), so the API is protected.

### P2 — Upload endpoint accepts arbitrary bucket names from user input
**File:** `app/api/upload/route.ts` line 13
```ts
const bucket = (formData.get('bucket') as string) || 'listings';
```
An authenticated user can upload to any Supabase storage bucket by passing an arbitrary `bucket` field. If Supabase RLS does not restrict write access to specific buckets, users could potentially overwrite files in other buckets (e.g., `avatars`, `documents`, `admin-assets`).

**Recommendation:** Whitelist allowed bucket names server-side:
```ts
const ALLOWED_BUCKETS = ['listings', 'avatars', 'itineraries'];
const requestedBucket = formData.get('bucket') as string;
const bucket = ALLOWED_BUCKETS.includes(requestedBucket) ? requestedBucket : 'listings';
```

### P2 — Provider authorization relies on RLS, not explicit ownership checks in some paths
The `app/api/whatsapp/send/route.ts` does correctly verify `conv.provider_id === provider.id` (line 44). The `app/api/discount-codes/route.ts` correctly scopes to `creator_id` via DB query. The `app/api/bookings/route.ts` GET scopes to `guest_email === user.email`. These are acceptable.

## Input Validation Issues

### Good — Most routes use Zod
The following routes validate input with Zod before use:
- `app/api/bookings/route.ts` — `createBookingSchema`
- `app/api/discount-codes/route.ts` — `createCodeSchema`
- `app/api/discount-codes/validate/route.ts` — `validateSchema`
- `app/api/ruta/checkout/route.ts` — `bookingRequestSchema`
- `app/api/ruta/tracker/ping/route.ts` — `trackerPingSchema`

### P2 — `app/api/stripe/checkout/route.ts` — `successUrl`/`cancelUrl` accepted from user input
**File:** `app/api/stripe/checkout/route.ts` lines 13-17
```ts
const { bookingId, successUrl, cancelUrl, discountCode } = body as { ... }
```
The caller-supplied `successUrl` and `cancelUrl` are passed directly to Stripe's `checkout.sessions.create` (line 69-71). Stripe does validate that redirect URLs are registered in the Stripe dashboard, but this still allows an open redirect if a malicious Stripe-registered domain is used. The caller should not control redirect URLs; they should be constructed server-side from `bookingId`.

### P2 — `app/api/ruta/geocode/route.ts` — `country` parameter not validated
**File:** line 9
```ts
const country = request.nextUrl.searchParams.get('country') || 'VE'
```
The `country` value is interpolated directly into the Google Places API URL (line 22): `components=country:${country.toLowerCase()}`. An attacker can set `country=us` to use your API quota to search for US places. Should be hardcoded to `'VE'` since this is a Venezuela-only platform.

### P1 — `app/api/whatsapp/config/route.ts` — no Zod validation on PUT body
**File:** `app/api/whatsapp/config/route.ts` lines 61-79. The PUT handler reads `body = await request.json()` and then manually filters allowed keys via an allowlist array. This is reasonable but `phone_number_id`, `sentiment_threshold`, `value_escalation_usd`, etc. are not type-validated. A provider could submit `sentiment_threshold: -999` or `response_delay_ms: 999999999` to cause unexpected behaviour.

### VERIFIED — WhatsApp webhook has no SQL injection risk
Supabase uses parameterized queries throughout. No raw string interpolation found in any DB query.

### VERIFIED — `lib/ai/search/route.ts` escapes SQL LIKE wildcards correctly
`get_safety_info` tool call line 113: `region.replace(/[%_]/g, '\\$&')` before `.ilike()`.

## SSRF Risks

### Good — `app/api/places/photo/route.ts` has SSRF guard
**File:** lines 19-21
```ts
if (!/^places\/[^/]+\/photos\/[^/]+$/.test(photoName)) {
  return new Response('Invalid photo reference', { status: 400 });
}
```
The regex validates that the photo reference matches the Google Places format before constructing the URL. No arbitrary URL fetching.

### P2 — `app/api/pms/[...path]/route.ts` — path segments not sanitized
**File:** lines 131-132
```ts
const pmsUrl = `${PMS_API_URL}/${path}${request.nextUrl.search}`;
```
`path` is the URL wildcard from `[...path]` and `PMS_API_URL` is `http://localhost:3001`. An authenticated provider could request `/api/pms/../../../etc/passwd` if path traversal is not handled by the PMS microservice. Since `PMS_API_URL` is an internal service URL and the path is constructed from the Next.js route, this is low risk in practice (the OS will normalize traversal), but should be validated.

### Good — no other routes proxy arbitrary user-supplied URLs

## CSRF Issues

### Low risk — Next.js App Router provides inherent CSRF protection
App Router routes are same-origin by default. Mutation routes use POST/PATCH/DELETE with JSON bodies (`Content-Type: application/json`), which browsers do not send via cross-site form submissions without explicit CORS allowance.

### P3 — No explicit CSRF token for auth flows
Login/register use Supabase's built-in auth which handles CSRF via the Supabase PKCE flow. No additional tokens are needed.

### P3 — No `SameSite=Strict` on admin cookie
**File:** `app/(admin)/admin/layout.tsx` line 47
```ts
document.cookie = `${name}=${...}; SameSite=Lax${secure}`;
```
`SameSite=Lax` is acceptable but `SameSite=Strict` would be more secure for admin cookies.

## Rate Limiting Gaps

### P0 — `app/api/whatsapp/send/route.ts` — no rate limit
The provider "send message" endpoint has no rate limiting. A compromised provider account or scripting could spam guests at no rate cap, triggering Meta account bans.

### P1 — `app/api/bookings/route.ts` POST — no rate limit
Any unauthenticated (or authenticated) user can call POST `/api/bookings` in a tight loop. At minimum this generates email spam (Resend sends a confirmation per booking) and fills the booking store with noise. At worst it could exhaust Stripe API quota.

### P1 — `app/api/stripe/checkout/route.ts` — no rate limit
POST to `/api/stripe/checkout` creates a Stripe Checkout session per call. Stripe charges for API calls, and abuse could exhaust rate limits or generate billing surprises.

### P1 — `app/api/places/autocomplete/route.ts` and `app/api/ruta/geocode/route.ts` — no rate limit
Both proxy to Google Places API with no rate limiting. Abuse could exhaust the $200/month free tier quickly.

### Good — AI routes are rate-limited
`/api/ai/search`, `/api/ai/fill-itinerary`, `/api/ai/optimize-itinerary`, `/api/ai/suggest-stops`, and `/api/itineraries/conversation` all call `rateLimit(getClientIp(request), 10)` (10 req/min per IP).

### Caveat — In-memory rate limiter resets on serverless cold start
**File:** `lib/api/rate-limit.ts`. The `requests` Map is module-level in-memory state. On Vercel serverless, each function instance has its own Map. Under load, multiple instances run in parallel and each has an empty Map, effectively multiplying the rate limit by the number of instances. For production, replace with a Redis/Upstash-backed rate limiter.

### Good — WhatsApp webhook has per-phone rate limiting
`app/api/whatsapp/webhook/route.ts` lines 39-50: 20 messages per minute per phone number. Same in-memory caveat applies but is mitigated by the low volume of inbound messages per number.

## WhatsApp Webhook Security

### VERIFIED — HMAC-SHA256 signature validation is present and correct
**File:** `app/api/whatsapp/webhook/route.ts` lines 20-35 (signature verification function) and lines 101-104 (enforcement).

The `verifyWebhookSignature()` function:
1. Reads `META_APP_SECRET` from server env (fails closed if unset — line 23-26)
2. Reads raw body text before JSON parsing (line 101) — correct, prevents body mutation attacks
3. Computes `sha256=<hmac>` and compares with `crypto.timingSafeEqual()` — correct, prevents timing attacks
4. Returns `false` (and triggers 403) on any failure

**This is properly implemented.**

### VERIFIED — Webhook token verification uses hashed storage
**File:** lines 68-93: The `hub.verify_token` is compared against the hashed stored value via `hashToken()`. Auto-migrates plaintext tokens to hashed on first use.

### P2 — Plaintext WhatsApp access token stored as DB fallback
**File:** `lib/whatsapp/token.ts` lines 33-34 and `app/api/whatsapp/config/route.ts` lines 104-128. When Supabase Vault is not available, the WhatsApp Business API access token is stored in plaintext in the `posada_whatsapp_config.access_token` column. A database breach would expose live Meta access tokens. The Vault migration path is present but not enforced.

### P2 — `META_APP_SECRET` missing causes full webhook rejection, not silent acceptance
This is actually the **correct** fail-closed behaviour. If `META_APP_SECRET` is not set, all inbound webhooks return 403. The error log message is clear.

## Priority Issues

### P0 (Critical — exploitable vulnerabilities)

**P0-1 — Client-side-only admin layout gate (bypassable in 5 seconds)**
- File: `app/(admin)/admin/layout.tsx`
- The SHA-256 hash `1967e7...` is visible in the JS bundle. Set the cookie manually in DevTools to bypass the gate. The admin UI renders fully (showing listing data, bookings, providers, etc.) even without a real Supabase admin session.
- Fix: Enforce auth check in a server component or Next.js middleware. Do not trust client-side cookie checks for access control.

**P0-2 — Hardcoded PMS bridge secret fallback**
- File: `app/api/pms/[...path]/route.ts` line 6
- `'vav-bridge-dev-secret-2026'` will be used as the PMS auth secret in any environment that does not set `PMS_BRIDGE_SECRET`. The secret is committed to the repo and effectively public to anyone with source access.
- Fix: Remove the fallback. Throw if `PMS_BRIDGE_SECRET` is not set.

### P1 (High — security weaknesses)

**P1-1 — Missing rate limiting on booking creation, Stripe checkout, and Google Places proxy**
- Files: `app/api/bookings/route.ts`, `app/api/stripe/checkout/route.ts`, `app/api/places/autocomplete/route.ts`, `app/api/ruta/geocode/route.ts`
- No per-IP rate limiting means these endpoints can be abused to exhaust Stripe API quotas, generate email spam, or consume Google Places billing.

**P1-2 — No rate limiting on `app/api/whatsapp/send/route.ts`**
- File: `app/api/whatsapp/send/route.ts`
- Compromised provider account can send unlimited WhatsApp messages, risking Meta account ban.

**P1-3 — `app/api/whatsapp/config/route.ts` PUT has no Zod validation**
- A provider can submit numeric fields with extreme values (`sentiment_threshold: -1`, `response_delay_ms: 9999999`) since only the key names are filtered, not their types or ranges.

**P1-4 — Upload endpoint accepts arbitrary Supabase bucket name**
- File: `app/api/upload/route.ts` line 13
- Any authenticated user can upload files to arbitrary storage buckets. RLS may not protect all buckets uniformly.

**P1-5 — `STRIPE_WEBHOOK_SECRET` used with non-null assertion without explicit check**
- File: `lib/stripe/server.ts` line 77
- Missing env var causes unhandled exception rather than graceful 503.

### P2 (Medium — defense in depth)

**P2-1 — `successUrl`/`cancelUrl` accepted from client in `/api/stripe/checkout`**
- File: `app/api/stripe/checkout/route.ts` lines 13-17
- Caller controls redirect URLs. Should be built server-side from `bookingId`.

**P2-2 — `country` parameter in geocode endpoint not restricted to VE**
- File: `app/api/ruta/geocode/route.ts` line 9
- Attacker can query Google Places for any country, consuming quota without geographic restriction.

**P2-3 — Plaintext WhatsApp access token stored as DB fallback**
- Until Vault is enforced, live Meta access tokens are exposed in Supabase plaintext column.

**P2-4 — PMS proxy does not validate path segments**
- File: `app/api/pms/[...path]/route.ts` line 132
- Path traversal sequences are forwarded to the PMS microservice.

**P2-5 — RUTA dispatch layout is client-side only**
- File: `app/(ruta)/ruta/dispatch/layout.tsx`
- UI access can be faked; API is protected but dispatch UI can be rendered by forging Supabase session.

**P2-6 — In-memory rate limiter is per-serverless-instance**
- File: `lib/api/rate-limit.ts`
- Effective rate limit multiplied by number of Vercel function instances. Replace with Upstash/Redis.

### P3 (Low — best practices)

**P3-1 — Admin cookie uses `SameSite=Lax` rather than `SameSite=Strict`**
- File: `app/(admin)/admin/layout.tsx` line 47

**P3-2 — Cron endpoint (`/api/ruta/cron/expire-zelle`) open if `CRON_SECRET` is not set**
- File: `app/api/ruta/cron/expire-zelle/route.ts` lines 9-13
- If `CRON_SECRET` is empty/unset, the check passes (`cronSecret` is falsy and the `&&` short-circuits). The endpoint is unauthenticated. This only expires Zelle rides so impact is limited, but could be used to cancel valid pending rides.

**P3-3 — Mapbox public token should be restricted by domain in Mapbox dashboard**
- `NEXT_PUBLIC_MAPBOX_TOKEN` is browser-visible by design but must have domain restrictions configured via the Mapbox token scopes dashboard to prevent quota theft.

**P3-4 — `ADMIN_HASH` constant in client bundle enables offline password cracking**
- Even if P0-1 is fixed, the hash being in the bundle allows offline brute-force. The admin password should never appear on the client side in any form.
