# A3 — UX Gap Analysis: Airbnb-Level Patterns
**Date:** 2026-04-20
**Scope:** VZ Explorer tourist-facing product versus Airbnb/Booking.com/Viator/GetYourGuide
**Prerequisite reports read:** D1 (Tourist Core), D2 (Booking Funnel), D3 (Auth + Accounts)

---

## Executive Summary

VZ Explorer is structurally more advanced than most niche destination platforms: it has a custom availability calendar, a price breakdown at checkout, a QR code on confirmation, and a functional review API. These are above average. But the platform falls short of the Airbnb conversion bar in five systemic areas that compound each other:

1. **Zero personalization signals** — no "recently viewed", no named wishlists, no "similar to places you saved", no behavioral recommendations. The platform treats every session as a first visit.
2. **Trust signals stop at "Verified"** — no response rate, no acceptance rate, no "Member since", no "Guests love this place" badge, no response time SLA displayed per provider.
3. **No in-platform messaging** — "Messages" in the nav routes users to WhatsApp. There is no inbox, no thread history, no booking-linked conversation record.
4. **Post-booking experience is a dead end** — the confirmation page has no "View my trips" CTA, no pre-arrival timeline, no check-in instructions link, and no local tips. The user is abandoned after payment.
5. **Mobile gallery and booking CTA are not native-quality** — the lightbox has no swipe gesture, the booking widget is below the fold on mobile with no sticky CTA, and the gallery shows no room labels.

## 1. Trust & Social Proof Gaps

### What exists
- `is_verified` badge on `ListingDetail` (line 171) — renders "Verified" Badge next to business name
- Safety badge (`SafetyBadge`) showing region risk level
- Star rating + review count in the listing header
- `is_featured` badge

### What is missing vs Airbnb

**A. Provider response rate and response time are not collected or displayed.**
Airbnb shows "Response rate: 99% · Response time: within an hour" directly below the host name. On VZ Explorer, `ListingDetail.tsx` shows only the business name, a Verified badge, a star rating, and a description. There is no `response_rate`, `response_time`, or `acceptance_rate` field on the `providers` table schema (not present in the select query at `listing/[slug]/page.tsx:59`). For a high-anxiety destination like Venezuela, response time is a primary trust signal — it tells the user that a real human will respond before they commit.

**Recommendation:** Add `response_rate` (%), `avg_response_hours`, and `acceptance_rate` (%) to the providers table. Display under the host name with icons: `Clock` for response time, `CheckCircle` for acceptance rate. These can be computed from `guest_bookings` records.

**B. "Host since" / "Member since" date is absent.**
Airbnb and GetYourGuide show how long a host has been on the platform. VZ Explorer shows `provider.description` only. `providers.created_at` exists in the DB but is never surfaced on the listing page. For a market-trust-building phase (VZ Explorer is new), displaying "Joined March 2025 · 12 bookings" is meaningful social proof.

**Recommendation:** Add a "Member since {monthYear}" line below the host description, derived from `provider.created_at`. Optional: add a total bookings count (aggregated from `guest_bookings`).

**C. No "Guests love this place" or achievement badge.**
Airbnb introduced "Guest Favorite" — a badge awarded to the top 10% of listings based on rating and booking volume. GetYourGuide uses "Award Winner." These social proof shorthand signals reduce cognitive load: the user doesn't need to read all 47 reviews to know this is a good choice.

VZ Explorer has `is_featured` (manually set by admin) but no algorithmically-derived badge based on guest behavior. There is a `recommendation_count` field on itineraries but no equivalent on listings.

**Recommendation:** Introduce a "Traveler's Pick" badge awarded to listings with `rating >= 4.5 && total_reviews >= 10`. Display it in the listing card alongside the existing `is_featured` treatment. This can be computed on-the-fly without a DB column change.

**D. Review count in listing card is absent on `BrowseListingCard`.**
`BrowseListingCard.tsx` does show `review_count` in a star row (via `StarRating`), but the count is visually de-emphasized. `ListingCard.tsx` shows `(N reviews)` inline. Airbnb shows "(47)" immediately after the star — the number is part of the trust hierarchy on the card itself. The pattern should be consistent between the two card implementations.

**E. "Photo verified" or "Real photos" badge is absent.**
Airbnb recently introduced photo quality signals. For Venezuela specifically, where travelers have high skepticism, a "Verified Photos" badge (even if manually applied by the ops team) would reduce abandonment on the listing detail page.

## 2. Review System

### What exists
- `ReviewSection.tsx` — well-built component with star selector, body textarea, image attachments, submit flow
- `review.provider_response` field exists and is displayed (line 170-175 in `ReviewSection.tsx`) — provider can respond to reviews
- Review gating: `canReview` prop only becomes true when a `bookingId` is passed, preventing review spam
- API: `POST /api/listings/:id/reviews` — functional
- Zero-review empty state is appropriate

### Critical gaps

**A. `canReview` is never passed for scraped/onboarded listings.**
`listing/[slug]/page.tsx:145` — `<ListingDetail listing={listing} reviews={[]} />`. No `canReview` or `bookingId` is passed. `ListingDetail.tsx:197` passes `canReview` and `bookingId` down only when `canReview` is truthy from props. For the Supabase path (line 76), these are also never set: `<ListingDetail listing={...} reviews={...} />`. The write-review button never appears for any listing, making the review system effectively read-only.

**B. There is no post-stay review trigger.**
Airbnb's review system works because it sends a review request email 24 hours after checkout. VZ Explorer's booking confirmation API (`/api/bookings/route.ts:276`) sends a confirmation email but there is no scheduled job, cron trigger, or webhook that sends a post-stay review request. The review system is available but guests are never prompted to use it.

**C. Reviews are unidirectional.**
The `ReviewSection` supports `provider_response` (host replies to guest review) but there is no system for providers to review guests. Airbnb's bidirectional review system, where both parties review each other simultaneously, is a core trust mechanism — it deters bad actors on both sides. VZ Explorer's review flow is guest-only. For a destination with legitimate security concerns, the ability for providers to flag problematic guests is a safety feature as much as a trust feature.

**D. Review sorting and filtering are absent.**
Airbnb lets users sort reviews by "Most recent" and filter by "Travel type" (solo, couple, family). VZ Explorer renders all reviews in a flat list ordered by `created_at DESC`. With no sort or filter, a listing with 20 reviews from 3 years ago and 2 recent ones will show the 3-year-old reviews most prominently.

**E. No category breakdown in rating summary.**
Airbnb breaks down ratings into: Cleanliness, Accuracy, Check-in, Communication, Location, Value. VZ Explorer shows a single average score with a star row. The `Review` type has only `rating` (int) and `body`. Sub-category ratings would require a schema change but are worth noting as a significant trust differentiator.

**Recommendation (P1 — fix immediately):** Wire `canReview={true}` and `bookingId={booking.id}` in the listing server component when the authenticated user has a `completed` booking for that listing. The logic: query `guest_bookings` for `{guest_email, listing_id, status: completed}` — if found, pass the booking ID to `ListingDetail`.

## 3. Search & Discovery Quality

### What exists
- AI-powered freeform search via `SearchOverlay` + `useSearch` hook with streaming responses
- Region and category filter in `ExploreClient`
- Debounced text search (300ms)
- `useRecentlyViewed` hook that persists to Supabase when authenticated
- Recently Viewed rendering in `trips/page.tsx`
- `useFavorites` hook with Supabase persistence

### Gaps

**A. Recently viewed is only shown on the Trips page — not at the top of Explore.**
Airbnb's primary browse surface shows "Continue where you left off" and "Recently viewed" before any algorithmic content. On VZ Explorer, `useRecentlyViewed` data is collected but only surfaced in `trips/page.tsx` (bottom of the page, below all tabs). A returning user at `/explore` sees no continuity. The hook is already implemented — the data just needs to be rendered at the top of the explore page for authenticated users.

**B. No named wishlists / collections.**
Airbnb's wishlist system ("Heart" → "Save to [list name]") is a core engagement loop. VZ Explorer has `useFavorites` (a flat "favorites" boolean per listing) and a `FavoriteButton` component. But there is no concept of named collections ("Los Roques Trip", "Budget Caracas", "Honeymoon Ideas"). All saves go into one undifferentiated pile visible in the Trips > Saved tab.

This matters beyond UX: named wishlists are a planning behavior signal that improves conversion. When a user creates a named wishlist, intent is high. Booking.com uses this pattern aggressively.

**C. No "Similar to places you've saved" or behavioral recommendations.**
VZ Explorer has no recommendation engine. The `itineraries` feed shows creator content, but the `explore` grid is sorted by `listing.created_at` or a basic sort. There is no "Because you saved Los Roques, you might like..." surface anywhere in the product. This requires ML/vector similarity to do properly, but a simpler heuristic (same region + similar category as recently viewed items) could be implemented with a raw SQL query and surfaced as a "More from this region" strip on the explore page.

**D. The search overlay has no structured date/destination/guest picker.**
Airbnb's search bar is a structured three-panel widget: Where / When / Who. Each field is independent and drives filtering. VZ Explorer's search is a single AI freeform query that returns streaming prose ("Here are some options in Merida..."). This is delightful for inspiration but poor for intent-matching. A user who types "Los Roques, Aug 10-15, 2 people" gets an AI response — but the Explore grid is not filtered to that destination + date range. The freeform search and the structured browse grid are disconnected systems.

**Recommendation (P2):** Add a "Where / When / Who" structured search widget to the homepage hero and the top of `/explore`. This widget should filter the `ExploreClient` grid (or construct a URL param like `?region=los-roques&checkin=2026-08-10&checkout=2026-08-15&guests=2`). The AI overlay search can remain as a secondary "ask anything" mode.

## 4. Pricing Transparency

### What exists
- `PriceDisplay` component in `BookingForm.tsx` shows the price prominently at `size="xl"` at the top of the sidebar card
- Full price breakdown (subtotal + 12% service fee + total) shown at Review step (step 3) and Payment step (step 4)
- "per person / night" label is present

### Gaps

**A. The total price (including service fee) is not visible before the user starts the booking flow.**
Airbnb's listing page shows the per-night rate AND "Total before taxes" in the sticky sidebar, before the user clicks any button. VZ Explorer's sidebar shows `$X / per person / night` but the service fee is invisible until step 3 of 4 in the booking flow. A user who sees "$120/night" and expects $120 will encounter $134.40 at the review step — a 12% surprise. This creates distrust even though the math is fair.

**Recommendation (P1):** In `BookingForm.tsx` step 1 (dates), show an estimated total range below the calendar: `"Estimated total with fees: $XX — $XX depending on stay length"`. Once dates and guests are selected (before the user advances to step 2), show the live total inline: `"$134.40 total (including $14.40 service fee)"`. This transparency increases rather than decreases conversion.

**B. The cancellation policy is in the sidebar but not shown before the booking flow begins.**
`ListingDetail.tsx:260-265` shows `listing.cancellation_policy` in a small sidebar card below the booking form. But this renders below the main booking widget, which is already below the fold on mobile. The policy is visible only after significant scroll on mobile. Airbnb shows the cancellation policy label (Flexible / Moderate / Strict) directly on the listing header next to the price.

**Recommendation (P1):** Move the cancellation policy badge to the listing header section, inline with the existing SafetyBadge and category badges. A small pill: `🔄 Flexible cancellation` or `⚡ Free cancellation` (Viator's label for flexible policies) is high-signal, low-cost real estate.

**C. The service fee percentage is not explained.**
The Review step shows "Service fee (12%): $14.40" but there is no tooltip or link explaining what the service fee covers. Airbnb provides a help link: "Why do I pay a service fee?" VZ Explorer users new to the platform may distrust an unexplained 12% charge.

## 5. Photo Experience

### What exists
- `ImageGallery.tsx` — 2-up grid layout (1 large left + 2 small right), with a `+N` overlay for additional photos
- Full lightbox: Escape to close, arrow key navigation, dot navigation, `aria-modal`, `ImageOff` fallback
- Focus trap is NOT implemented in the lightbox (unlike the AuthModal which has one)
- `next/image` used for `ListingCard` gallery; raw `<img>` used in `BrowseListingCard` (D1 P1 issue)

### Gaps

**A. No swipe gesture support in the mobile lightbox.**
The `LightboxOverlay` component handles `ArrowLeft`/`ArrowRight` keyboard events (line 161-168) but has zero touch event handling. On mobile, the user sees arrow buttons (`ChevronLeft` / `ChevronRight`) at 44px each — functional but not native-feeling. Airbnb's mobile gallery is swipeable. The dot navigation dots at the bottom (`w-3 h-3 rounded-full`) are 12px — too small for touch target even with the `min-w/h` not applied to these dots.

**Recommendation (P1):** Add `onTouchStart` / `onTouchEnd` handlers to `LightboxOverlay`: track `touchStartX`, on `touchEnd` if delta > 50px, call `onPrev()` or `onNext()`. This is a ~15-line addition.

**B. No photo room labels / captions.**
Airbnb labels gallery photos: "Bedroom", "Kitchen", "View", "Bathroom". This allows users to quickly find specific photos relevant to their stay. VZ Explorer's `Photo` type has an `alt` field (used for screen reader text) but there is no caption or room-label rendered in the gallery thumbnail grid or in the lightbox. `photos.map(url => ({ url, alt: listing.name }))` — all alt text is the listing name, not the photo subject.

**Recommendation (P2):** Extend the `listing_photos` schema with a `caption` field (nullable string). Display it as a short label overlay on gallery thumbnails and as text below the image in the lightbox. Providers should be able to set captions in the `ListingWizard` photo step.

**C. The 2-up grid is capped at 3 images visible.**
Airbnb's gallery shows 5 images in a 2-2-1 grid before the "Show all photos" modal. VZ Explorer shows 1 large + 2 small = 3 images before the `+N` overlay. GetYourGuide shows a horizontal scrollable strip with all photos. Three images is below the norm. For accommodation listings especially, travelers want to see the bathroom, bedroom, and outdoor areas before deciding to book.

**Recommendation (P2):** Expand to 5-image layout (1 large left + 4 small right in a 2x2 grid) when 5+ photos are available. Add a prominent "Show all photos" button that opens the lightbox. This is a standard pattern on all top travel platforms.

**D. No "Add cover photo" prompt for listings without photos.**
When `cover_image_url` is null (scraped listings with broken photo URLs, as noted in D2), `ListingDetail.tsx` skips the gallery entirely (`allImages.length > 0`). The listing renders with no photos at all — just a white gap where the gallery was. There is no prompt, placeholder, or "Photos coming soon" state.

## 6. Communication Patterns

### What exists
- `app/(tourist)/messages/page.tsx` — a "Messages" page that generates WhatsApp deep-links per booking
- Quick-reply templates (Check-in time, Directions, Breakfast, Early arrival)
- WhatsApp contact button on `ListingDetail` when `provider.whatsapp_number` is present
- Provider messaging via `app/(provider)/dashboard/messages/` (WhatsApp AI brain)

### Gaps

**A. There is no in-platform messaging system.**
This is the most significant structural gap relative to Airbnb. Every "Message Host" action on VZ Explorer routes the user to WhatsApp (or fails silently when `provider_phone` is null — D3 P1 bug). The "Messages" page is a WhatsApp launcher, not an inbox.

This means:
- Conversation history is not stored in the platform
- The ops team cannot monitor or moderate guest-provider communication
- Booking context is lost (which booking are they discussing?)
- Users without WhatsApp cannot contact hosts
- Providers who haven't set up WhatsApp are unreachable before booking

Airbnb's messaging system is deeply integrated: every message is linked to a listing/booking, the thread is pre-initialized with booking details, and the platform retains conversation history for dispute resolution. Booking.com, GetYourGuide, and Viator all have in-platform messaging.

**Recommendation (P2 — significant effort):** Build a lightweight in-platform message thread: `messages` table (`id`, `booking_id`, `sender_id`, `sender_role`, `body`, `created_at`). UI: a `/messages/[bookingId]` route with a chat thread. WhatsApp can remain as an alternative channel but should not be the primary one.

A lower-effort interim pattern: pre-compose a structured message template and store it in Supabase even if delivery still goes via WhatsApp — this at least gives ops visibility into pre-booking communication volume.

**B. No "Contact Host" button before booking.**
Airbnb prominently surfaces "Contact host" at the bottom of the listing description, BEFORE the user commits to booking. This is used for questions ("Is the location walkable to the beach?", "Can you accommodate dietary restrictions?"). VZ Explorer has no pre-booking contact mechanism other than a WhatsApp button that appears only when `provider.whatsapp_number` is set. For scraped listings, there is no contact method at all.

**C. No message notification badge in the nav.**
The `MobileTabBar` has no unread message count badge. The nav `Messages` tab has no indicator when a provider has replied. Since messages are WhatsApp-based, this is technically impossible to implement without WhatsApp Business API webhook support — but it highlights the architectural problem.

## 7. Cancellation & Refund Clarity

### What exists
- `cancellation_policy` field on `Listing` type — plain text string
- `ListingDetail.tsx:260-265` — displayed in a small box in the sidebar below the booking widget
- `getRefundEstimate()` function in `bookings/[id]/page.tsx:40-45` — computes refund based on days until check-in
- Cancellation option in `trips/page.tsx` BookingCard — inline confirm, no modal

### Gaps

**A. Cancellation policy is below the fold on mobile and not visible before starting the booking flow.**
As noted in Section 4B, the policy card renders below the sticky booking widget in the desktop sidebar. On mobile, it renders after scrolling through reviews (step 2 of the main content column). A user who has already entered guest details and is at the Review step (step 3) may not have seen the cancellation terms — they discover them only after viewing the full detail page.

Airbnb shows the cancellation policy summary ("Free cancellation before Aug 8") prominently on the listing header and repeats it at the booking confirmation step.

**B. The refund estimate is not shown in the trips-list cancel flow (D2 P2-4).**
`getRefundEstimate()` exists in `bookings/[id]/page.tsx` but is not wired to `trips/page.tsx` where the inline cancel button lives. Users cancelling from the trips list (the most common cancellation path) don't see what they'll get back before confirming. This creates anxiety and may prevent legitimate cancellations or cause post-cancellation support requests.

**C. Cancellation policy is free-text, not structured.**
`OnboardingWizard.tsx:70` defines `cancellation_policy` as `'flexible' | 'moderate' | 'strict'` — but when written to the listing, `ListingDetail.tsx` receives it as a plain string and renders it as-is. There is no standardized icon, color, or tooltip that maps "flexible" to "Free cancellation up to 48h before" — the user sees whatever the provider typed in their onboarding form.

**Recommendation (P1):** Normalize the three policy types to human-readable summaries:
- Flexible: "Free cancellation up to 48 hours before check-in"
- Moderate: "50% refund if cancelled 5+ days before check-in"
- Strict: "No refund within 7 days of check-in"

Display this as a colored badge in the listing header. Repeat the policy at step 3 (Review) of the booking form.

## 8. Post-Booking Experience

### What exists
- Confirmation page with: status card, QR code, booking reference (copyable), booking details, ICS calendar download, WhatsApp share, "Build a trip around this" upsell card
- Email confirmation sent via Resend (plain text, correct content)
- Booking detail page (`/bookings/[id]`) with full timeline and refund estimate

### Gaps

**A. No "View my trips" CTA on the confirmation page (D2 P1-3).**
The primary post-booking action is missing. `ConfirmationClient.tsx:151-157` only shows "Back to home" (`href="/"`). After completing a booking, the user's most natural next action is to view their upcoming trips or continue planning around this booking. The confirmation page should have a prominent `<Button asChild><Link href="/trips">View my trips</Link></Button>` as the primary CTA.

**B. No pre-arrival timeline or check-in instructions.**
Airbnb sends: (1) booking confirmation email, (2) pre-arrival reminder 3 days before check-in with check-in instructions, (3) day-of-travel check-in guide with door codes / meeting point / host contact. VZ Explorer sends one email: the booking confirmation (plain text). There is no scheduled pre-arrival email, no in-app check-in instructions card, and no "What to bring" or "Meeting point" section on the booking detail page.

The `listing.meeting_point` field exists in the Listing type (`meeting_point: null` in the scraped mapping at `listing/[slug]/page.tsx:138`) but is never displayed to the user at the listing detail or booking detail pages.

**Recommendation (P2):** Add a `PreArrivalCard` component to `bookings/[id]/page.tsx` that shows: meeting point, languages spoken, what to bring (from `listing.includes`), and provider WhatsApp. This card should conditionally render 7 days before `check_in` — always visible after booking, highlighted in a different color in the final week.

For the email side: add a `/api/cron/pre-arrival` route (already has a cron setup pattern via `/api/ruta/cron/`) that queries bookings with `check_in = today + 2 days` and sends a pre-arrival reminder email. The `TODO` comment in `/api/ruta/cron/expire-zelle/route.ts:48` ("Send expiration notification emails") suggests this pattern is intended.

**C. No local tips or curated recommendations for the booked destination.**
After booking, Airbnb shows "Things to do near {listing}" and "Guidebook from {host}". VZ Explorer has rich editorial content (region pages, safety zones, itineraries) but none of it is surfaced post-booking. A user who just booked a posada in Merida should see: "Your host recommends...", "Safety info for Mérida", "5-day itinerary for the Andes". These cross-links already exist as pages — they just need to be surfaced on the confirmation and booking detail pages.

**D. Notification system exists but is not surfaced in the UI.**
`/api/notifications/route.ts` — a fully functional read-notifications API exists (GET unread, PATCH mark-read). But there is no bell icon in the Navbar or MobileTabBar that shows unread count. No component in the tourist layout reads from `/api/notifications`. The backend is built; the frontend surface is missing.

## 9. Mobile-Native Patterns

### What exists
- `MobileTabBar.tsx` — 5-item bottom nav (Home/Explore/Plan/Trips/Profile), `md:hidden`, 44px min touch targets, `safe-area-inset-bottom` support, `aria-current` per tab
- `pb-16 md:pb-0` padding in tourist layout to account for tab bar
- Mobile category chips in ExploreClient

### Gaps

**A. No sticky bottom booking CTA on mobile (D2 P1-1).**
`ListingDetail.tsx:207` — `lg:sticky lg:top-24`. On mobile, the BookingForm renders at the very bottom of the page after photos, description, host info, and reviews. There is no fixed "From $X / night — Reserve" bar at the bottom of the screen. This is the single highest-impact conversion gap. Airbnb, Booking.com, and every major OTA have a fixed bottom booking bar on mobile.

**Recommendation (P1):** Add to `ListingDetail.tsx`:
```tsx
{/* Mobile sticky booking CTA */}
<div className="fixed bottom-16 left-0 right-0 z-30 bg-background border-t p-4 flex items-center justify-between lg:hidden">
  <div>
    <PriceDisplay price={listing.price_usd} size="sm" />
    <span className="text-xs text-muted-foreground">per person / night</span>
  </div>
  <Button onClick={() => bookingRef.current?.scrollIntoView()} className="px-6">
    Reserve
  </Button>
</div>
```

**B. The photo lightbox has no swipe gesture support (covered in Section 5A).**
This is the mobile-native equivalent of the desktop arrow key navigation. Not having swipe support in a photo gallery is a jarring mobile experience that breaks the illusion of a native app.

**C. The MobileTabBar is not rendered on the `/map` page (D1 P1).**
The map page (`app/map/page.tsx`) is outside the `(tourist)` layout group. It renders no navigation. Mobile users on the map page cannot navigate back without the browser back button.

**D. No bottom sheet for filters on mobile.**
Airbnb's mobile filter experience opens as a bottom sheet with smooth spring animation. VZ Explorer's `FilterOverlay` is centered on desktop and at `items-end` on mobile (a simple bottom alignment, no sheet animation, no drag-to-dismiss). The filter overlay is functional but lacks the native-quality spring animation and drag affordance that makes Airbnb's mobile feel premium.

**E. The "Plan" tab in MobileTabBar routes to `/plan` — which may not have a consumer-facing page.**
`/plan` is linked from the hero and the mobile tab bar. Its content and completeness as a consumer planning surface is not audited here but should be verified.

**F. No haptic feedback integration.**
Native Airbnb uses `UIImpactFeedbackGenerator` for button taps. On web, this is approximated with `navigator.vibrate(10)` on key interactions (booking confirmed, item saved). VZ Explorer has no haptic feedback. Low-priority but contributes to the "native app feel."

## 10. Onboarding & First-Time User Experience

### What exists
- Landing page (`app/page.tsx`) — a real landing page (not a redirect), with: crossfade hero, "How it works" section, stats strip, editorial picks grid, category strip (Airbnb-style icon row), AI planner teaser, social proof section, CTA banner
- The "How it works" 3-step section explicitly addresses the trust problem ("Every provider is locally verified")
- Both CTAs in the hero ("Plan my trip with AI", "Explore Venezuela") are clear and differentiated

### Gaps

**A. No structured "Where / When / Who" search widget on the landing page.**
The hero CTAs are binary: either start an AI conversation (`/plan`) or browse all listings (`/explore`). There is no structured search input on the homepage. Airbnb's homepage has a single focal element: the search bar. Booking.com has the same. GetYourGuide has a destination + date search. VZ Explorer's homepage asks users to self-select into two different flows without a middle path.

A tourist who knows they want "Los Roques, 4 days in August" has no direct path from the homepage. They must go to `/explore`, filter by region, and find date-based availability separately.

**B. The category strip uses `/library/category/...` links which are stale.**
`app/page.tsx:69-76` — all 8 CATEGORIES point to `/library/category/{slug}`. The DESIGN.md decision log (2026-04-19) merges `/library` into `/explore`. These links are stale and will 404 or redirect incorrectly. The "Where to begin" editorial picks (line 47-65) also link to `/library/region/...`. The homepage is cross-linking to the deprecated route.

**C. No personalized hero for returning logged-in users.**
Airbnb shows different homepage content for logged-in vs. anonymous users. Logged-in users see wishlists, recently viewed, and personalized picks. VZ Explorer renders the exact same landing page for everyone. The `useAuth` hook is available but `app/page.tsx` is a server component that doesn't use it.

For logged-in users, the hero could show: "Welcome back, [name]. Continue planning your Merida trip?" with a link to their last-viewed listing or draft itinerary.

**D. No value proposition for the specific "Venezuela" concern.**
The landing page's social proof section (line 388-428) addresses trust with: "Verified providers", "Safety zones", "WhatsApp support". These are correct. But they are generic trust signals. The most common objection from the target user (international adventure traveler discovering Venezuela on social media) is: "Is it actually safe? Am I going to get robbed?". The safety value prop is present but not specific enough. Airbnb dedicates hero copy to "Host Guarantee" and "$3M AirCover insurance" — specific, quantified guarantees. VZ Explorer's claims are benefit-language without concrete backing ("Real businesses, real insurance, real accountability" — what insurance specifically?).

**E. No guided onboarding flow for first-time users.**
After logging in or registering for the first time, the user is immediately routed to `/` (the landing page) or `/` via `router.push('/')`. There is no "Tell us about your trip" onboarding modal, no interest survey, no "What brings you to Venezuela?" flow. Platforms like Viator and GetYourGuide use a 2-3 screen onboarding that captures trip intent and personalizes the subsequent browse experience. VZ Explorer treats first-time users identically to returning users.

**F. The search bar on the landing page is absent entirely.**
The `SearchBar` and `SearchOverlay` components exist but are only accessible from the `Navbar` (which triggers the overlay on click). There is no inline search bar on the landing page itself. For a user who arrives from a social media post that says "This beach in Venezuela is incredible" — the first action they want to take is search for it, not read "How it works." The search bar should be the hero's primary interactive element.

## What VZ Explorer Does Well (Do Not Break)

These patterns meet or exceed Airbnb standards and should be preserved:

- **Availability calendar** — custom-built, per-day pricing, range hover preview, ARIA grid. Genuinely good.
- **QR code on confirmation** — practical for a destination with connectivity concerns.
- **ICS calendar download** — underrated feature that increases show-up rate.
- **Safety integration throughout** — SafetyBadge on listing cards, safety zones in browse, region safety on detail. No competitor does this.
- **AI trip planner** — unique and genuinely useful. The streaming UX is production quality.
- **WhatsApp-native communication** — correct for Venezuela (WhatsApp penetration ~95%). The WhatsApp integration is right even if the in-platform messaging is missing.
- **Zelle / USDT / Pay on Arrival payment options** — appropriate for the Latin American context where Stripe card penetration is lower.
- **`auth-modal` focus trap** — production quality accessibility that most travel sites don't have.
